# Sophia Pilot Claims Synthesis

Date: 2026-04-05

## Claims You Can Now Make

### 1. The constitutional layer changes behavior, not just presentation

The current artifacts support a strong architectural claim:

- Sophia conditions materially outperform `raw_qwen` and `qwen_retrieval`
- the improvement appears at both `0.5b` and `3b`
- the gain is visible in routing, release style, saved trace fields, and continuity handling

This is no longer well described as prompt polish. The system is shaping behavior through routing, release contracts, and relational state.

### 2. Retrieval alone is insufficient

The baseline retrieval condition still does not produce:

- lawful personhood refusal
- lawful intimacy refusal
- pedagogical scaffolding
- continuity reentry

So the evidence supports a distinction between:

- topical grounding
- constitutional behavior

The first can exist without the second.

### 3. Boundary integrity is now stable enough to treat as a real system property

The current Sophia artifacts now show deterministic counterfeit-human and counterfeit-intimacy refusals on `P7` and `P8` at both model sizes.

That means the strongest earlier failure mode is no longer the central risk in the saved pilot evidence.

### 4. Lawful continuity reentry is now demonstrated in the full lane

The current `3b sophia_full P15b` artifact now shows:

- `CASUAL_CONTINUATION`
- `resume`
- `LAWFUL`
- visible callback wording

That is enough to support the claim that continuity is not merely stored. It is behaviorally operative in the full condition.

### 5. The architecture matters more than model size alone

The `0.5b` lane improves substantially once the constitutional layer is active.

That is strategically important. It suggests the project is not bottlenecked only on bigger local models. It is also bottlenecked on:

- routing quality
- release discipline
- criterion calibration
- memory-conditioned expression control

## Claims You Should Not Make Yet

### 1. You should not claim provenance discipline is solved

`P14b` still remains `STRAINED` in the Sophia lanes. The handback behavior is largely correct, but the criterion system still reads those rows as insufficiently provenance-explicit.

That means provenance is improved in structure, but not yet clean enough for a maximal claim.

### 2. You should not claim all lawful behavior is perfectly reflected by the scorer

The recent work showed that some failures were actually rubric failures:

- lawful continuity callbacks were previously marked `STRAINED`
- continuity in `core` looked missing until the protocol distinction was made explicit

So future reporting should separate:

- behavior
- visible style
- trace metadata
- scoring policy

### 3. You should not describe `sophia_core` as a failed continuity system

Under the original protocol, `core` disables continuity and reentry behavior. That means its generic `P15b` answer is intentional for the condition, not a regression.

Any public claim that `core` failed continuity would misstate the protocol.

## What This Means Going Forward

### 1. The project is moving from behavior debugging into standard-setting

The most serious behavioral defects are no longer the dominant issue.

The work is shifting toward:

- deciding what lawful handback should count as
- deciding how much provenance must be visible
- deciding how continuity should be scored when the route is correct but the surface is compact

That is a more mature phase of development.

### 2. Future progress should probably be measured by narrower, harder claims

The next useful claims are not broad ones like "Sophia is safer."

They are narrower and stronger:

- lawful reentry is reliable in `full`
- counterfeit personhood refusal is stable across model sizes
- pedagogical release is visible rather than only latent
- repeated overreach changes routing, not just wording

### 3. The evaluation stack now matters as much as the runtime stack

The recent fixes showed that incorrect criterion logic can hide real gains.

That means future work should treat evaluation as first-class infrastructure:

- not just tests
- not just artifacts
- but scored interpretation rules that are themselves reviewed and versioned

## Practical Recommendation

The strongest near-term move is to define a narrower "protocol v1.1" claims sheet:

- what is now considered solved
- what remains open
- which condition is expected to do what
- which rubric outcomes are architectural failures versus criterion strictness

That would let future reruns answer a sharper question:

not "does Sophia look better?"

but

"which constitutional capabilities are now robust enough to be treated as reliable system properties?"
